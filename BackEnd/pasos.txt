## 📋 Tus Tareas por Sprint

### Sprint 1: Diseño y Estructura
* **Definir la API:** Diseñar qué rutas (endpoints) necesita el grupo. [cite_start]Por ejemplo: `GET /activos` (precios actuales) y `POST /chat` (pregunta al RAG)[cite: 640, 641].
* [cite_start]**Configuración Segura:** Preparar el sistema para leer variables de entorno (`.env`) usando **SSM Parameter Store** de AWS para no subir contraseñas al Git[cite: 93, 492, 597].
* [cite_start]**Esquema de Base de Datos:** Ayudar al responsable de IA a definir cómo se guardarán las trazas de las consultas[cite: 575, 643].

### Sprint 2: Desarrollo del Motor RAG
* [cite_start]**Integración con FAISS:** Crear la lógica que conecte FastAPI con el índice vectorial que cree tu compañero de IA[cite: 593, 638, 734].
* [cite_start]**Lógica de Negocio:** Programar el flujo: el usuario pregunta -> tú buscas en los datos de Spark/IA -> tú generas la respuesta final[cite: 559, 765].
* [cite_start]**Sistema de Moderación:** Implementar un filtro (basado en listas de palabras o modelos ligeros) para bloquear mensajes tóxicos o fuera del ámbito financiero[cite: 537, 574, 647].

### Sprint 3: Conectividad y Logging
* [cite_start]**CORS:** Configurar la API para que el compañero de Frontend pueda conectarse desde el WordPress o la web institucional[cite: 539, 561].
* [cite_start]**Registro de Interacciones:** Crear un sistema de **Logging anonimizado** para guardar qué pregunta la gente (sin guardar quién es) para las métricas finales[cite: 538, 595, 702].

### Sprint 4: Calidad y Pruebas
* [cite_start]**Pruebas de Funcionalidad:** Verificar que la API responde rápido (latencia baja) y no se cae con consultas ambiguas[cite: 469, 801, 805].
* [cite_start]**Documentación API:** Usar la autodocumentación de FastAPI (Swagger) para que tus compañeros sepan cómo usar tus rutas[cite: 565, 876].
